ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ К ПРОГРАММНОМУ ОБЕСПЕЧЕНИЮ ТРЕНАЖЕРА «ТП-СИМУЛЯТОР»
Приоритет
1. Общие требования к системе
Система должна быть построена по клиент-серверной архитектуре. +
Сервер должен поддерживать одновременное проведение одной учебной сессии. +
Клиентское приложение должно предоставлять различные интерфейсы в зависимости от роли пользователя. +
Все изменения в состоянии учебной обстановки (пожар, силы, средства) должны синхронизироваться между сервером и
клиентами в реальном времени. +
2. Модуль ролевого управления и связи
Система должна предоставлять следующие ролевые модели (рабочие места):
Руководитель занятий (Администратор/Сервер). +
Диспетчер (Д). +
Руководитель тушения пожара (РТП). +
Начальник штаба (НШ), Командир отделения (КО), Ствольщик (С), Водитель (В). -
Должен быть реализован модуль симплексной голосовой связи между ролями в рамках одной учебной сессии. +
Связь должна работать по принципу Push-To-Talk (PTT) с захватом канала. Только одна роль может передавать голосовой
сигнал в единицу времени. +
На интерфейсе всех участников должен отображаться индикатор текущего говорящего (например, `[ГОВОРИТ:
ДИСПЕТЧЕР]`). -
Система должна вести автоматическую аудиозапись и текстовый лог (стенограмму) всех переговоров сессии.
Для передачи формальных приказов система должна предоставлять интерфейс для отправки структурированных
сообщений (например, выбор техники, действия, параметра из выпадающих списков), которые автоматически отображаются
на схемах других ролей.

3. Модуль конфигурации и сценариев (Руководитель занятий)
Интерфейс должен позволять загружать в систему растровые или векторные схемы объектов (планы зданий, генпланы) с
привязкой масштаба (метров на пиксель). +
Должна быть предусмотрена библиотека типовых объектов с заданными тактическими характеристиками (степень
огнестойкости, категория по взрывопожарной опасности, конструктивные особенности согласно Справочнику). -
Руководитель должен иметь графические инструменты для задания на схеме:
Очага пожара (точка). +
Зоны пожара (полигон) с возможностью динамического изменения. +
Зона температурного воздействия для введения ручных стволов +
Зон задымления. +
Мест нахождения условных пострадавших.
Должна быть возможность ввода и изменения исходных данных сценария:
Метеоусловия (ветер, температура). +
Время суток. +
Состояние систем противопожарного водоснабжения (исправны/неисправны). +
Должна быть возможность ввода внештатных вводных в ходе занятия (отказ техники, изменение направления ветра,
обрушение конструкции, увеличение площади пожара). +
Должна быть функция «оперативного скачка времени» — ускоренного моделирования развития обстановки на заданный
промежуток.

4. Модуль моделирования пожара и динамики обстановки
Система должна рассчитывать развитие пожара на основе физической модели, используя данные из Справочника:
-
Расчет линейной скорости распространения (Vл) в зависимости от типа горючей нагрузки. +
Автоматический расчет площади (Sп) и периметра (Pп) пожара для различных геометрических форм (круговая, угловая,
прямоугольная) +
Учет влияния метеоусловий (ветра) на скорость и направление развития.
Модель должна реагировать на действия обучаемых:
При вводе стволов на тушение скорость распространения должна снижаться, а площадь пожара — стабилизироваться или
уменьшаться, в зависимости от достаточности сил и средств.
При выполнении мероприятий по защите (охлаждению) соседних объектов модель должна оценивать их эффективность. +

5. Модуль управления силами и средствами (Диспетчер, РТП)
В базе данных должна быть перечислена пожарная техника и вооружения с тактико-техническими характеристиками (ТТХ)
согласно Справочнику
Для техники: тип (АЦ, АЛ, АСА), запас воды/пенообразователя, тип и длина рукавов в боевом расчете, численность
экипажа/расчета. +
Для вооружения: тип ствола (РС-50, РС-70, Дельта, ГПС-600), расход воды/раствора, расход стволов. Запас Огнетушащих
веществ на автомобиле, запас рукавов на автомобиле-
Рабочее место Диспетчера:
Получает вводную о пожаре.
в последующем, при получении информации от руководителя тушения пожара, рисует схему расстановки сил и средств +
На основе справочных данных и алгоритмов расчета (Справочник) формирует рекомендацию по количеству и типу сил и
средств для высылки по повышенному номеру (рангу) пожара. +
Осуществляет «высылку» расчетов , равной расчетному времени следования (Справочник). +
Ведет журнал учета высылки и поступления информации.

6. Модуль документации и анализа (Начальник штаба)
Рабочее место должно в реальном времени получать и графически отображать на собственной схеме все распоряжения
РТП, переданные голосом и/или структурированными командами. +
Должен автоматически вестись оперативная карта пожаротушения, фиксирующая расстановку сил, позиции стволов,
рукавные линии на временной шкале. +
По окончании сессии система должна автоматически генерировать отчет, включающий: -
Хронологию событий и переговоров.
Графики изменения площади пожара, расхода огнетушащих веществ.
Расчетные показатели: требуемый и фактический расходы, достаточность сил и средств, оценку временных параметров.
Протокол с анализом соответствия действий обучаемых положениям Боевого устава и Справочника.

8. Требования к безопасности и тестированию на уязвимости
SQL-инъекция: Все поля ввода данных должны быть проверены на невозможность внедрения произвольного SQL-кода. +
Межсайтовый скриптинг (XSS): Все пользовательские данные, отображаемые в интерфейсе, должны быть проверены на
невозможность выполнения вредоносных скриптов. +
Слабые учетные данные: В системе должны отсутствовать предустановленные учетные записи со стандартными паролями. +
Незакрытые сетевые порты: Серверное и клиентское окружение не должно иметь открытых портов, не используемых
функционалом приложения. +
Отсутствие/неверная настройка SSL/TLS: Весь обмен данными между клиентами и сервером должен быть защищен
актуальными версиями криптографических протоколов. +
Утечка чувствительной информации: В корневых каталогах или репозитории должны отсутствовать файлы, содержащие
пароли, резервные копии БД, приватные ключи. +
Устойчивость к некорректным данным: Решение должно корректно обрабатывать все входящие данные: как валидировать и
очищать пользовательский ввод, предотвращая сбои, так и обеспечивать безопасную десериализацию, исключая
выполнение произвольного кода. +
Устойчивость контроля доступа: Решение должно строго ограничивать пользователя рамками его роли, предотвращая
несанкционированный доступ к чужому функционалу через манипуляции с запросами (IDOR). Одновременно механизмы
сессии должны быть защищены от перехвата, фиксации и повторного использования. +
Использование устаревших версий программного обеспечения, фреймворков, библиотек +
